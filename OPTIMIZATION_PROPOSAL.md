# InsureScope 第一性原理检视与优化建议

> 检视日期：2026-07-07 ｜ 当前版本：2.2.1 ｜ 数据量：26 条资讯 / 15 个信源

---

## 一、第一性原理：项目的本质目标

剥离所有实现细节（HTML/JS/CSS、CloudStudio 部署、主题系统），这个项目要回答一个根本问题：

> **如何让保险从业者，以最低成本，持续获取高价值的行业资讯？**

价值公式：

```
用户价值 = (信息质量 × 新鲜度 × 覆盖面) / 获取成本
```

**当前方案在分母（获取成本 = 人工撰写 + 整包重部署）上很高，在分子（新鲜度、覆盖面）上很低** —— 这是与"聚合"本质的根本矛盾，也是所有优化建议的起点。

---

## 二、当前方案依赖的隐含假设（及破绽）

| 隐含假设 | 现实检验（基于真实代码） | 结论 |
|---------|------------------------|------|
| "人工逐条撰写资讯可行" | 26 条数据横跨 2024-06 至 2026-07；最近更新 2026-07-02，距今天（7/7）已 **5 天零更新** | ❌ 不可持续 |
| "纯静态零依赖最优" | 每次更新需改 `data.json` + 整包重部署 CloudStudio，更新摩擦高 | ❌ 与"频繁更新"诉求冲突 |
| "展示层是价值核心" | UI 已成熟（参考 InsureAI/aihot），边际收益递减 | ⚠️ 不再是瓶颈 |
| "`data.json` 统计准确" | `days` 统计混用 0-10 / 0-100 两制；`source_health` 是写死假数据 | ❌ 数据失真 |
| "信源健康度可信" | `sources[].last_update` 全是 2024 年；`source_health` 无采集脚本支撑 | ❌ 虚假指标 |

---

## 三、按第一性原理排序的优化建议

### P0 — 回归"聚合"本质（最高优先级，解决根本矛盾）

| # | 建议 | 第一性原理依据 | 可行性 |
|---|------|--------------|--------|
| **1** | **建立自动化采集管道** | "聚合"的前提是自动获取；人工手敲不可持续。可复用 InsureAI 远程仓库的采集脚本 + GitHub Actions 定时运行，或做本地一键采集工具 | 中 |
| **2** | **数据与前端解耦部署** | 更新资讯不应重部署整站。让 `data.json` 独立托管（远程 URL / 自动提交），HTML 引用之，使"发布资讯"成本趋近于零 | 高 |

### P1 — 修复数据信任基础（低成本、高信任收益）

| # | 建议 | 说明 |
|---|------|------|
| **3** | 统一评分标准 | `days["2026-06-12"].avg_score = 7.7`（0-10 制遗留）vs `days["2026-07-02"].avg_score = 84.3`（0-100 制），必须统一为单一标准 |
| **4** | 修复 `days` 统计逻辑 | 当前把全部 20 条历史数据（含 15 条 2024 年旧数据）都算作 2026-06-12 当天新增，统计错误 |
| **5** | 移除 / 真正实现 `source_health` | 当前 `eastmoney/iachina/baidu/akshare` 是硬编码假数据，项目无采集脚本，应删除或接真实采集流 |
| **6** | 刷新 `sources[].last_update` | 全部为 2024 年，给人"停更"印象，损害信任；应动态更新或标注"评分快照日" |
| **7** | 自动 id 与日期规范化 | id 手动维护易冲突（已出现 1-15 与 16-26 分段）；日期格式 `Z` 与 `+08:00` 混用 |

### P2 — 放大产品价值

| # | 建议 | 说明 |
|---|------|------|
| **8** | SEO / 预渲染 | SPA + `fetch` 加载导致搜索引擎无法抓取正文，资讯站流量基础缺失；建议预渲染或 SSR |
| **9** | 真实反馈闭环 | 反馈页为静态表单，无后端；建议对接 GitHub Issues API 或轻量表单服务 |
| **10** | 来源验证与去重实跑 | 已有 Levenshtein 去重框架，需对接真实采集流才能真正生效 |

### P3 — 工程优雅性（权衡项）

| # | 建议 | 说明 |
|---|------|------|
| **11** | 单文件拆分 | `index.html` 1896 行三合一（HTML+CSS+JS），长期演进困难；可拆 `css/`、`js/`（代价：失去单文件便利性） |

## 四、关键数据问题清单（基于 data.json 实测）

1. `days["2026-06-12"].total = 20`，但其中 15 条真实发布日期为 2024-06，**统计口径错误**
2. **评分标准分裂**：早期 0-10 制遗留数据 vs 新 0-100 制数据并存
3. `source_health` 四项为**硬编码假数据**，无任何采集脚本支撑
4. `sources[].last_update` 全部为 **2024 年**，严重过时
5. 多条记录 `source_url` 为空（如 id 9、10）
6. 多条记录 `research_topic` 为空（如 id 11、14、20）
7. 日期时区格式不统一（`Z` vs `+08:00`）

---

## 五、建议执行顺序

```
第 1 步（本周）：P1 数据失真修复        → 低成本、立刻恢复信任
第 2 步（核心）：P0 采集管道 + 解耦部署  → 解决"聚合"本质矛盾
第 3 步（放大）：P2 SEO + 反馈闭环       → 让价值被看见、被反哺
第 4 步（可选）：P3 单文件拆分           → 工程优雅性，按需推进
```

---

## 六、一句话总结

> **InsureScope 的 UI 已超前于它的"内容引擎"。真正的瓶颈不在前端，而在"如何持续、自动、低成本地生产可信数据"——这是第一性原理指出的唯一值得投入的方向。**
